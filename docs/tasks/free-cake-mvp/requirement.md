## 业务目标

- **目标用户**: 村镇 10km 内居民（C端） + 总部运营团队（B端） + 门店执行人员
- **核心问题**: 乡镇蛋糕市场缺乏标准化运营和数字化管理
- **成功指标**: 参与人数≥500 | 刷票拦截≥90% | 结算准确100% | 核销率≥85% | 转付费≥5%

## 验收标准 (AC)

### AC-01: AI 出图与参赛
- **Given**: 用户已完成手机号验证且定位在 10km 服务范围内
- **When**: 用户提交蛋糕设计需求并选择参赛
- **Then**: 系统在 30 秒内生成 5 张蛋糕效果图，用户可选择 1 张发布参赛

### AC-02: 投票与风控
- **Given**: 活动处于投票开放阶段
- **When**: 用户对参赛作品投票
- **Then**: 系统记录投票，每日限投 3 票，命中风控策略的票先冻结后复核

### AC-03: 自动结算
- **Given**: 活动投票截止且有至少 100 个参赛作品
- **When**: 系统自动结算
- **Then**: 按有效票排序生成 Top100 获奖名单，同步生成奖励订单与核销码

### AC-04: 核销领取
- **Given**: 获奖用户持有有效核销码且在领取时效内
- **When**: 用户到门店扫码核销
- **Then**: 系统校验手机号+核销码，确认领取成功，核销码失效不可重复使用

### AC-05: 活动管理
- **Given**: B 端运营人员登录管理后台
- **When**: 创建新活动并配置活动规则
- **Then**: 活动进入 draft 状态，可配置赛区、时间、奖品规则、投票规则，支持模板复用

### AC-06: 风控审核
- **Given**: 风控系统检测到异常投票行为
- **When**: B 端运营人员在风控审核台查看异常事件
- **Then**: 系统展示风险分布、设备/IP 聚类视图，支持一键冻结/解冻和批量扣减异常票

## 当前实现状态 (2026-04-29)

### 服务端 (50+ 端点, 9 DB migrations)
- **认证**: 手机号+验证码登录、微信 OpenID 登录、手机号绑定、JWT 刷新/注销/黑名单
- **活动管理**: CRUD + 状态机流转 + 规则配置 + 模板复用 + banner_url
- **参赛与 AI**: 作品提交、AI 生成脚手架（mock URL）、模板选择
- **投票与排行**: 投票含风控校验、Redis 排名缓存、RANK() 窗口函数排行、user_name/is_winner
- **开奖结算**: 手动+自动结算（winner_record + reward_order + redeem_code 联动生成）
- **订单**: 免费订单自动生成、付费下单（price_config 查询 + 活动状态/作品状态校验 + DB 事务防竞态）、订单详情/列表
- **支付**: 微信 JSAPI 支付集成、prepay_id + PrepayParams 返回、回调验签脚手架（dev 模式降级）、退款 API 脚手架
- **核销**: 到店扫码核销、Redis 分布式锁防重复、重发核销码
- **库存**: CRUD + 出入库流水 + 安全阈值预警
- **排产**: 任务状态机 (pending→in_progress→completed/paused/error/cancelled)
- **风控**: 多维度联合判定、异常事件持久化（含 show 端点）、批量冻结/扣票
- **后台调度**: 活动自动流转、订单超时关闭、自动结算
- **中间件**: RBAC 鉴权、IP 限流、公开端点限流、请求 ID、Prometheus 指标、安全头、体量限制
- **对账**: 支付-订单对账、投票计数漂移、库存余额漂移
- **报表**: 汇总 + 对账 + 日/周/月报（含 region_id 过滤）
- **单元测试**: 46 个全通过
